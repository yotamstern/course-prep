Both the server and client send a finish+ack [F.] that symoblizes the end of the connection and that there is no more data to send/recieve. Then the server sends a [.] that acks the closing request of the client.
The flow is: Server [F.] -> Client [F.] -> Server [.]

The 3 step handshake is 3 requests long because you need to ack both client and server and request a syn for both sides. It cannot be possible in less than 3 requests.

2. The TCP lifecycle start with establishing a connection between two sides: server and a client (most times). The client sends a SYN request to the server, server responds with a SYN and an ACK, then the client ACKs the connection and then they can communicate. Each time one side communicates with the other (wants to send a HTTP payload) it does PUSH+ACK request. The ACK is mandatory for each request during the session after the three way handshake to acknowledge that the side has acknowledged all of the previous requests. The receiving side then ACKs the request. When the disconnection happens (teardown), the client sends a FIN + ACK (ack here is mandatory becuase of the earlier rule), the server sends a FIN+ACK aswell, then the client ACKs and then the connection closes.
