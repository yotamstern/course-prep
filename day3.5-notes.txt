I learned from this lesson that there are multiple ways of debugging errors with crontabs running a script. It can be a cron problem (file doesnt exist), a problem inside the script (syntax error) or a permission problem.
Each one could be found in a different place. For example: The syntax error is found in the error file that we appended the stderr from the script into since cron ran the script but the problem was the script itself. 
The permissions problem was found in the same file aswell since it couldnt run the script therefor the stderr was put into the log file.
The "file is not found" error was found in the journalctl -u cron. Since the problem was on the cron level.

FHS: 

/var/log - directory with dynamic log files for different parts of the system.

/var/log/syslog - log file that logs processes and services in the system. Good use for debugging scripts that run for example.
 
/var/log/auth.log - logs the different connections made on the machine that required authentication, sudo actions, and security events

/etc/cron.d - a cron directory for system level cronjobs. Usually for cronjobs of specific programs and not for user cronjobs like what I did earlier. 

/etc/cron.daily -a dir that executes the scripts on it at a specific time each day. The system decides at what time.
